1. WordPress vs Hugo — porównanie technologii

WordPress to naturalne pierwsze skojarzenie. Ale w naszym scenariuszu — kilkadziesiąt (docelowo kilkaset) portali zarządzanych automatycznie przez Claude Code — WordPress wprowadza problemy, których statyczny generator stron nie ma.

Koszty i utrzymanie

Na jednym hostingu za ~400–500 zł/rok (z 15 GB miejsca) da się postawić nawet 100 instalacji WordPress — treści tekstowe nie zajmują dużo. Sam hosting nie jest więc problemem. Problem zaczyna się gdzie indziej:

Aspekt WordPress (~100 stron na hostingu) Hugo + Cloudflare Pages
Hosting (10 portali) ~35–42 zł/mies. (1 hosting, ~500 zł/rok) 0 zł (Cloudflare Free)
Hosting (50 portali) ~35–42 zł/mies. (1 hosting wystarczy) 0 zł
Hosting (200 portali) ~70–125 zł/mies. (2-3 hostingi) 0 zł (ew. $20 za Pro)
Aktualizacje WP + wtyczki Ciągłe — co tydzień coś wymaga update'u. Przy 50 stronach to kilka godzin tygodniowo Brak — nie ma co aktualizować
Bezpieczeństwo WordPress = najczęściej atakowany CMS. Każda instalacja to ryzyko (brute force, exploity wtyczek, SQL injection). Jeden zhakowany WP może zainfekować cały serwer Zerowa powierzchnia ataku — statyczne pliki HTML na CDN. Nie ma bazy danych, nie ma PHP, nie ma czego atakować
Backupy Trzeba konfigurować per instalacja (baza + pliki). Przywracanie = import SQL + pliki Git = backup. Pełna historia zmian. git revert przywraca cokolwiek w sekundy
Core Web Vitals Wymaga optymalizacji: cache, CDN, optymalizacja obrazów, lekki motyw. Elementor/Divi = katastrofa Idealny wynik out-of-box (czyste HTML z CDN)
Automatyzacja (Claude Code) REST API — wymaga autoryzacji, obsługi błędów sieciowych, wtyczek do custom fields. Każdy krok = potencjalny punkt awarii Zapis pliku .md + git push = strona live w 30 sek. Natywne środowisko Claude Code
Kluczowa różnica: hosting WordPress jest do ogarnięcia cenowo. Ale utrzymanie 50–200 instalacji WordPress (aktualizacje, bezpieczeństwo, backupy, optymalizacja prędkości) to dziesiątki godzin pracy miesięcznie. W modelu Hugo te problemy po prostu nie istnieją — bo nie ma bazy danych, nie ma PHP, nie ma wtyczek do aktualizowania.

❌ WordPress — problem z automatyzacją

Żeby Claude Code opublikował artykuł na WordPress, musi:

  1. Połączyć się przez REST API (token, autoryzacja)
  2. Stworzyć post (obsługa błędów sieciowych)
  3. Ustawić kategorię, tagi, autora (custom fields = wtyczka)
  4. Dodać Schema markup (wtyczka Yoast/RankMath)
  5. Sprawdzić czy się opublikowało (timeout? 500?)

Przy 200 stronach — 200 różnych API endpointów do obsługi.

✅ Hugo — natywna automatyzacja

Żeby Claude Code opublikował artykuł na Hugo, musi:

  1. Zapisać plik .md w folderze
  2. Wykonać git push

Koniec. Cloudflare automatycznie buduje i wdraża stronę. Schema, autorzy, kategorie — wszystko w szablonie. Przy 200 stronach — ten sam prosty flow.

2. Co to jest Hugo i jak działa

Hugo to generator stron statycznych (SSG). Bierze pliki tekstowe (Markdown) i szablony, i generuje gotową stronę HTML. Nie ma żadnej bazy danych, żadnego serwera aplikacyjnego, żadnego PHP.

Jak wygląda proces publikacji

1. Claude Code pisze artykuł jako plik tekstowy (.md)
2. Hugo generuje z niego stronę HTML (poniżej 1 sekundy)
3. Pliki trafiają na Cloudflare Pages (automatycznie po git push)
4. Strona jest dostępna na całym świecie przez CDN (300+ lokalizacji)

Czy Hugo nie jest przeładowane JavaScriptem?

Nie. Hugo generuje czyste HTML. Domyślnie na stronie nie ma ani jednej linijki JavaScriptu — chyba że sami zdecydujemy się coś dodać (np. analytics).
Technologia JavaScript na stronie Wpływ na SEO
WordPress + Elementor/Divi 500 KB – 2 MB JS Negatywny wolne ładowanie, opóźniony render
WordPress + lekki motyw 100–300 KB JS Neutralny
Hugo (domyślnie) 0 KB JS Pozytywny natychmiastowy render

Czy Google dobrze pozycjonuje strony Hugo?

Google nie rozróżnia technologii — ocenia efekt końcowy. A efekt końcowy strony Hugo to:

  • Błyskawiczny czas ładowania — idealne Core Web Vitals bez konfiguracji
  • Czysty, semantyczny HTML — bez zbędnego kodu i frameworków
  • Pełna kontrola nad SEO — Schema markup, meta tagi, struktura URL w szablonach
  • Zero problemów z renderowaniem JS — bo JS nie ma, bot Google dostaje gotowy HTML
Przykłady dużych stron na Hugo: dokumentacja Kubernetes, Let's Encrypt, 1Password Support, Cloudflare Docs — wszystkie rankują w Google na topowe pozycje.

Czy darmowy plan jest stabilny?

Tak. Cloudflare Pages Free to produkcyjny produkt, nie trial. Używają go tysiące firm. Cloudflare zarabia na wyższych planach (Enterprise) — Free tier to ich strategia pozyskiwania klientów. Nie jest zagrożony wycofaniem.

Kiedy warto dopłacić

PotrzebaPlanKoszt
Więcej niż 500 buildów/mies. lub buildy równoległePro$20/mies. (za cały account)
Zaawansowany WAF, Bot ManagementPro/Business$20–200/mies.
Dedykowany supportEnterprise$200+/mies.

Rekomendacja na start: plan Free. Przy 10–20 portalach i 5–7 artykułach dziennie zmieścimy się w limicie. Upgrade do Pro ($20/mies.) dopiero jak przekroczymy 500 buildów.

Backup — jak to działa

Repozytorium Git (GitHub) JEST backupem. Cały portal — każdy artykuł, szablon, konfiguracja — to pliki w repozytorium z pełną historią zmian.
  • Ktoś skasował artykuł? → git revert przywraca go w sekundę
  • Cloudflare padło? → Portal jest w GitHub, deploy na inny hosting w minuty
  • Chcemy wrócić do wersji sprzed tygodnia? → git checkout i gotowe

To lepszy backup niż WordPress, gdzie przywracanie oznacza import bazy SQL + pliki + modlitwa żeby wersje się zgadzały.

6. Pełny proces — od zera do działającego portalu

ETAP 0 — Przygotowanie infrastruktury jednorazowo, tydzień 1

Kto: Claude Code + Mateusz
  1. Szablon Hugo — responsywny design, Schema markup (Article, Person, FAQ, BreadcrumbList), strony autorów, sitemap.xml, robots.txt, Open Graph. Cel: 100/100 PageSpeed Insights
  2. Pipeline Python — moduły: research (Senuto API + SERP scraping), generowanie (Claude/GPT/Gemini API), weryfikacja (encje, unikalność), publikacja (git push), dystrybucja (Facebook API)
  3. Konfiguracja — konto Cloudflare, organizacja GitHub, repo-szablon

ETAP 1 — Segmentacja nisz tydzień 1–2

Kto: Mateusz (decyzje) + Claude Code (analiza)
  1. Eksport listy domen klientów z obecnymi kategoriami
  2. Claude Code analizuje i proponuje granularną taksonomię:
    Nie: „Dom i ogród" → Tak: „Szamba i oczyszczalnie", „Nawozy i środki ochrony roślin", „Meble ogrodowe"
  3. Mateusz zatwierdza listę nisz i ustala priorytety (pod największych klientów)

ETAP 2 — Research i planowanie klastrów 2–3 dni per portal

Kto: Claude Code (autonomicznie) → Mateusz (akceptacja)
  1. Seed keywords z Senuto API dla niszy
  2. Query fan-out: People Also Ask, related searches, long-tail z Senuto
  3. SERP scraping top 10 — jakie tematy, encje i formaty u konkurencji
  4. Clusterowanie fraz — grupowanie wg intencji → pillar pages + satelity
  5. Mapa encji per klaster (osoby, firmy, produkty, koncepcje)
  6. Harmonogram publikacji — co, kiedy, w jakiej kolejności
  7. Mateusz akceptuje mapę → start produkcji

ETAP 3 — Setup portalu 1 dzień per portal

Kto: Claude Code + Mateusz (domena + social)
  1. Mateusz rejestruje domenę
  2. Claude Code: klonuje szablon, konfiguruje branding (nazwa, kolory), tworzy 2–3 profile autorów, podpina do Cloudflare Pages
  3. Mateusz zakłada konta social media (FB, opcjonalnie LinkedIn)

ETAP 4 — Produkcja contentu ciągła, ~150 art./mies.

Kto: Claude Code (autonomicznie)

Dla każdego artykułu (cykl ~7 minut):

  1. Pobranie tematu z zatwierdzonej mapy klastrów
  2. Ekstrakcja docelowych encji
  3. Wygenerowanie artykułu — odpowiedni model AI (rotacja), unikalna persona autora, 800–4000 słów, pokrycie encji ≥ 70%, FAQ z PAA, 1–2 linki do autorytatywnych źródeł
  4. Weryfikacja: encje ✓, unikalność ✓, Schema ✓, linkowanie wewnętrzne ✓
  5. Zapis .md → git push → Cloudflare auto-deploy (30 sek.)
  6. Automatyczny post na FB/LinkedIn
  7. Log do arkusza (tytuł, URL, data, klaster, autor, status)

ETAP 5 — Dojrzewanie i monitoring ciągłe

Kto: Claude Code (monitoring) + Mateusz (decyzje)
  • Monitoring indeksacji (GSC) — czy artykuły wchodzą do indeksu
  • Monitoring pozycji (Senuto) — czy portal rankuje
  • Cytowania AI Overviews — ręcznie z Senuto (na start)
  • Po 90 dniach + 12 artykułów → flaga „dojrzała" → portal gotowy do linkowania

ETAP 6 — Realizacja linków po dojrzewaniu

Kto: Mateusz (matching) + Claude Code (realizacja)
  1. Fraza klienta wpada do systemu
  2. Matching tematyczny — dopasowanie do portalu z pasującą niszą
  3. Claude Code pisze artykuł kontekstowy z naturalnym linkiem do klienta
  4. Publikacja w standardowym flow
